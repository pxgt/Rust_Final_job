# SpecProbe 实验评估记录

## 1. 实验目的

验证 SpecProbe 能否在一个带有已知缺陷的 Web 项目上完成项目识别、需求解析、运行取证、问题生成和修复提案生成，并明确当前课程版本的能力边界。

## 2. 被测对象

- 项目：FocusBoard
- 路径：`demo/buggy-task-board`
- 技术栈：Node.js、原生 HTML/CSS/JavaScript
- 第三方依赖：无
- 需求文档：`demo/buggy-task-board/REQUIREMENTS.md`
- 已知缺陷基准：`demo/buggy-task-board/KNOWN_ISSUES.md`

## 3. 实验方法

执行：

```powershell
.\scripts\run-demo.ps1
```

脚本依次运行 `scan`、`requirements`、`ai`、`launch --dry-run`、`browser`、`review` 和 `propose`，报告保存到 `.specprobe/demo-reports/`。

随后使用浏览器在桌面宽度和 375px 宽度下进行人工复核，并操作输入框、复选框和筛选按钮。

## 5. 已知缺陷覆盖

| 缺陷 | 自动发现 | 人工浏览器复核 |
| --- | --- | --- |
| `/api/tasks` 返回 HTTP 500 | 是 | 是 |
| 空白任务仍被添加 | 否 | 是 |
| 已完成统计不更新 | 否 | 是 |
| 已完成筛选无效 | 否 | 是 |
| 新增任务刷新后丢失 | 否 | 是 |

当前版本对 5 个已知动态缺陷自动发现 1 个，动态缺陷覆盖率为 20%。这一结果符合当前基础执行器只做 HTTP 页面探测、尚未执行真实点击和输入的实现范围。

## 6. 浏览器复核结果

- 页面桌面布局正常，统计卡片、任务输入区和任务列表层级清晰。
- 375px 宽度下 `documentScrollWidth` 等于可用页面宽度，没有横向溢出。
- 控制台记录 `Task API returned HTTP 500`。
- 输入空格后任务数从 3 增加到 4，且没有校验提示。
- 勾选第二个任务后，完成样式任务数为 2，但“已完成”统计仍为 0。
- 点击“已完成”筛选后按钮进入激活状态，但仍显示 4 个任务。
- 新增“持久化验证任务”后刷新页面，该任务消失。

## 7. 结果分析

实验说明 SpecProbe 已经形成稳定的基础闭环：

1. 能从需求文档产生结构化测试计划。
2. 能识别项目启动方式并安全地进行 dry-run。
3. 能采集确定性的 HTTP 运行证据。
4. 能把失败证据转换为带严重度和证据编号的 Issue。
5. 能进一步生成修复提案、补丁预览和回归检查。

同时，实验也暴露出两个明确限制：

- `has_observable_result` 目前采用关键词规则，对部分已经具备可验证语义的需求仍可能给出保守提示。
- 浏览器执行器尚未执行点击、输入、DOM 断言、截图和网络监听，因此无法自动发现四个交互缺陷。

## 8. 后续改进

1. 接入 Playwright 后端，将浏览器动作计划映射为真实操作。
2. 捕获控制台错误、网络失败、截图和 DOM 断言。
3. 改进需求可观察性判断，减少规则型误报。
4. 增加用户审批持久化和真实 Git patch 应用。
5. 在更多项目类型上进行对照实验。
