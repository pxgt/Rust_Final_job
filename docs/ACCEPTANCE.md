## 暴露的质量调优项(LLM 输出波动)

真机验收的价值在于暴露了确定性测试之外的 LLM 稳定性问题,均为可迭代的调优点,非架构缺陷:

1. **断言类 selector 猜测导致误报**:REQ-008 猜测横幅为 `.error-banner`,实际是 `#api-banner`(横幅本身工作正常,首轮用 `text=` 断言即 PASS)→ 假阳性。**方向**:断言类 selector 也优先取自页面快照,或强制用文本断言而非猜测 class。
2. **筛选类缺陷需要 negative 断言原语**:DEMO-004 要检出"点击已完成后不该出现的未完成项仍在",需要 `expect_absent` / `expect_hidden` 动作(当前只有存在性断言)。**方向**:sidecar 增加否定断言原语。
3. **诊断在失败较多时过度合并**:失败 issue 增多时,诊断把 8 个 issue 归入单一根因并出现定位漂移(首轮 3 个独立诊断、server.js:47 精确;多失败轮次合并为 1 个、定位漂到 app.js)。**方向**:限制单诊断关联 issue 数,或按类别分组诊断。
4. **运行间波动**:相同输入(温度 0)下,prompt 微调即改变场景与检出(首轮 3/5、强化断言后 4/5),说明检出率应多次运行取稳定值,单次结果仅供参考。

## Phase 1.8 调优后复测(2026-07-08,--no-cache)

三项机制调优全部落地并验证有效:

- ✅ **否定断言原语**:新增 `expect_hidden`(sidecar `waitForSelector state:hidden`),筛选/隐藏类需求可用。
- ✅ **断言 selector 收敛**:REQ-008 之前把横幅猜成 `.error-banner` 的误报**已消除**——复测改用 `text=部分数据未同步`,正确 PASS。
- ✅ **诊断防过度合并**:从上一轮的 1 个合并诊断恢复为 3 个独立诊断,API 500 精确定位 `server.js:46`。

但同一轮 `--no-cache` 复测(每次重新生成场景)暴露了更本质的限制:**LLM 场景生成本身不稳定**,单次结果在 3~4/5 之间波动,并伴随不同的假阳/假阴:

- 漏步骤:REQ-001 生成的步骤 `fill` 后漏了 `click`,任务未添加导致断言失败(假阳)。
- 否定断言目标编造:REQ-005 对一个未先创建的任务名做 `expect_hidden`,元素本就不存在 → 立即"通过"(假阴,DEMO-004 仍漏检)。
- 断言强度回退:REQ-003 又退回弱断言(仅断言标签存在),DEMO-003 漏检。
- 断言过严:REQ-012 对 `h1` 断言精确文本导致假阳。

## 结论

Phase 1 架构目标(真需求解析 → 真起服务 → 真浏览器执行 → 真交互步骤 → 证据 → LLM 源码诊断)端到端验证成功;注入缺陷检出从基线 1/5 提升到 **单次 3~4/5 波动**,已达数量门槛。诊断源码定位准确(server.js:46/47 经核对无误)。

**关键判断**:检出率的进一步稳定提升已触及"纯 prompt 调优"的天花板——LLM 每次生成的场景步骤有内在波动(漏步骤、编造断言目标、断言强度不一),再多的 prompt 规则边际收益递减且易过拟合单个被测项目。真正的解法是**场景执行级修复回路**(ROADMAP 1.5 原设想的"生成→dry 执行→失败步骤带证据回喂 LLM 修正",当前仅做了静态 selector 校验,未做执行级修复),以及**多次采样取一致**。这是一块独立的硬骨头,投入产出上应权衡是否在进 Phase 2 前做。1.8 的三项机制改进本身是正确且必要的基础设施,予以保留。
