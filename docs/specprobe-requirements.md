# SpecProbe 课程版本需求说明

## 功能需求

- 用户必须可以通过命令行扫描一个项目目录，并显示项目技术栈、需求文档、源码语言和测试文件数量。
- 用户必须可以导入 Markdown 或 TXT 需求文档，并生成结构化需求列表。
- 系统应该为每条需求生成至少一条可验证的验收标准。
- 系统应该根据需求类别生成初始测试用例，包括测试步骤、期望结果和执行器建议。
- 系统应该支持 JSON 输出，供后续 AI 分析、测试执行器和报告界面读取。
- 系统可以识别需求中的模糊描述，并提示用户补充量化标准。
- 用户必须可以使用离线 Mock Provider 对需求解析结果生成 AI 增强建议。
- 系统应该为 AI 建议标注建议类型、严重程度、理由和关联需求编号。
- 系统应该在未配置云端 API key 时给出明确错误提示，而不是静默失败。
- 用户必须可以识别项目启动命令，并在受控超时时间内运行被测项目。
- 系统应该采集被测项目的 stdout、stderr、退出码、运行耗时和超时状态。
- 系统应该支持只检测启动命令但不执行的 dry-run 模式。
- 用户必须可以把结构化测试计划转换为浏览器动作计划。
- 系统应该支持浏览器执行器 dry-run 模式，展示将要打开的页面、交互动作和断言。
- 系统应该可以访问本地 `http://` 页面，并采集 HTTP 状态码、页面标题和正文摘要。
- 用户必须可以把需求质量、项目启动和浏览器探测证据汇总为结构化缺陷报告。
- 每个缺陷问题必须包含严重程度、问题类别、预期结果、实际结果、证据编号和修改建议。
- 系统应该为缺陷问题保留 pending、accepted、rejected、ignored 等审批状态。
- 用户必须可以根据缺陷问题生成修复提案。
- 每个修复提案应该包含关联问题编号、目标文件、修复步骤、风险提示和审批状态。
- 系统应该为可安全预览的问题生成 Git patch 风格的补丁预览。
- 系统应该为每个修复提案生成回归检查命令或人工验证步骤。
- 项目必须提供一个包含已知缺陷的 Web 演示项目，用于验证完整审查流程。
- 用户必须可以通过一个脚本运行扫描、需求解析、AI 分析、页面探测、综合审查和修复提案流程。
- 项目应该提供实验评估记录和课程报告初稿。

## 非功能需求

- 需求解析过程必须跳过代码块，避免把示例代码误识别为真实需求。
- 扫描项目时必须跳过 `.git`、`target`、`node_modules` 等依赖或构建目录。
- 在 Windows 环境下，系统应该能够处理 MSVC 编译环境和 `.cmd` 命令执行。
- Mock Provider 的输出必须稳定可重复，便于课程演示和回归测试。
- 运行被测项目时必须对日志中的 token、password、secret、api key 等敏感信息进行脱敏。
- 浏览器页面探测应该能够处理 chunked HTTP 响应，避免正文摘要包含传输编码噪声。
- 缺陷报告生成过程应该稳定可重复，相同输入和证据应得到相同的问题编号和证据编号。
- 修复提案生成过程应该稳定可重复，相同问题清单应得到相同的提案编号和回归检查编号。
- 演示项目应该不依赖第三方 npm 包，避免因网络或依赖安装导致课堂演示中断。

## 安全需求

- 系统必须默认只读分析被测项目，不应直接修改用户代码。
- 综合审查命令默认应该进行计划级审查，只有用户显式启用执行模式时才运行项目和页面探测。
- 修复提案命令必须只生成预览和验证清单，不应自动应用补丁。
- 后续生成修改建议时，系统必须以补丁形式展示，并等待用户审批。
