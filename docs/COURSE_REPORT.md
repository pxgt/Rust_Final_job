# 基于 Rust 与大语言模型的智能代码审查及自动化测试平台

> 课程阶段（0.8.0）报告初稿，历史文档。接手后的项目规划见 [ROADMAP.md](ROADMAP.md)。

## 摘要

AI Coding 工具能够快速生成大量代码，但宽泛需求容易导致生成结果停留在初稿状态，出现功能缺失、运行错误、交互不符合预期和测试不足等问题。面对规模较大的 AI 生成代码，普通用户往往无法有效定位和描述问题。

本项目设计并实现了 SpecProbe，一套使用 Rust 开发的、证据驱动的智能代码审查与自动化测试平台。系统能够扫描项目结构，解析 Markdown/TXT 需求文档，生成结构化验收标准和测试计划，识别并受控运行被测项目，采集进程日志和 HTTP 页面证据，再将失败证据转换为结构化 Issue 和修复提案。系统通过 `AiProvider` 抽象支持离线 Mock Provider、OpenAI 兼容接口和本地 Ollama，并坚持默认只读、执行受控、修改需审批的安全原则。

实验使用一个包含五类预置缺陷的任务看板作为被测对象。SpecProbe 成功识别 Node.js 技术栈、提取 12 条需求、生成 12 个测试用例，并稳定发现 HTTP 500 故障，生成 4 个 Issue、4 个修复提案和 9 个回归检查。实验也表明，当前基础浏览器探测器无法自动发现点击、输入和持久化类缺陷，后续需要接入 Playwright 深度执行器。

**关键词：** Rust；大语言模型；代码审查；自动化测试；证据链；AI Coding

## 1. 研究背景

### 1.1 问题来源

AI Coding 的输入经常只是较宽泛的产品构思，而不是完整的软件需求规格。模型会据此生成可以运行的初稿，但初稿通常缺少边界处理、错误反馈、完整交互流程和可重复测试。

### 1.2 现有使用方式的不足

- 只让模型阅读源码，缺少真实运行证据。
- 用户需要逐条描述问题，效率较低。
- 非专业用户难以定位大型生成代码中的具体缺陷。
- 模型可能直接修改代码，缺少审批和回归验证。

### 1.3 项目目标

SpecProbe 的目标是把需求、确定性测试、运行证据、AI 推理和用户审批组织成可追踪流程，而不是让大语言模型直接获得无限制执行和修改权限。

## 2. 需求分析

系统核心需求包括：

1. 项目技术栈和测试表面扫描。
2. 需求解析、质量提示和验收标准生成。
3. 可替换的 AI Provider。
4. 被测项目启动命令识别和受控执行。
5. 浏览器测试动作计划和页面证据采集。
6. Issue、证据链和审批状态。
7. 修复提案、补丁预览和回归检查。

非功能需求包括稳定可重复、JSON 可序列化、日志脱敏、默认只读和执行超时。

## 3. 总体设计

```text
需求文档 + 项目代码
        |
        v
Scanner -> Requirement Analyzer -> TestPlan
        |                         |
        v                         v
Project Adapter              Test Executor
        |                         |
        +---------- Evidence -----+
                      |
                      v
                Review / Issue
                      |
                      v
          Patch Proposal / Regression
                      |
                      v
                 User Approval
```

系统核心流程不绑定特定语言或框架。课程版本优先实现 Web 项目适配器，但数据模型和模块边界可以继续扩展到 CLI、API、桌面应用和移动应用。

## 4. 关键模块

| 模块 | 主要职责 |
| --- | --- |
| `scanner.rs` | 扫描技术栈、源码、需求文档和测试文件 |
| `requirements.rs` | 提取需求、验收标准、质量提示和测试计划 |
| `ai.rs` | AI Provider 抽象和离线 Mock 分析 |
| `runtime.rs` | 启动命令识别、超时、日志采集和脱敏 |
| `browser.rs` | 浏览器动作计划和基础 HTTP 页面探测 |
| `review.rs` | 证据归一化、Issue 和审批状态 |
| `remediation.rs` | 修复提案、补丁预览和回归检查 |
| `output.rs` | 终端和 JSON 报告输出 |

## 5. Rust 实现特点

### 5.1 类型安全的数据模型

项目使用结构体和枚举表达需求类别、证据类型、执行状态、问题严重度、审批状态和修复策略，避免在模块之间传递无法校验的松散字符串。

### 5.2 Trait 扩展机制

`AiProvider` 将 Mock、OpenAI 兼容接口和 Ollama 隔离在统一抽象之后。类似方式还可以继续扩展 `ProjectAdapter`、`TestExecutor`、`EvidenceCollector` 和 `ReportRenderer`。

### 5.3 错误处理

项目使用 `thiserror` 定义模块错误，使用 `anyhow` 处理 CLI 顶层错误，使错误原因能够保留并显示给用户。

### 5.4 安全设计

- 默认计划级分析，不主动执行项目。
- 子进程设置超时并在超时后终止。
- 日志中的 token、password、secret 和 API key 相关行会被脱敏。
- 修复提案只生成预览，不自动修改代码。
- 所有问题和补丁都预留用户审批状态。

## 6. 实验设计与结果

实验对象为 FocusBoard 任务看板，其中预置 API 500、空输入校验、统计更新、筛选和持久化五类缺陷。

主要结果：

- 提取 12 条需求并生成 12 个测试用例。
- 生成 15 条 Mock AI 建议。
- 正确识别 `npm run dev` 启动命令。
- 首页探测返回 HTTP 200，并正确提取页面标题。
- 故障 API 返回 HTTP 500，被转换为高严重度 `browser_failure` Issue。
- 综合生成 4 个 Issue、4 个修复提案、3 个补丁预览和 9 个回归检查。
- 29 个 Rust 单元测试全部通过，严格模式 Clippy 无警告。

完整实验表见 `docs/EXPERIMENT.md`。

## 7. 项目创新点

1. 将 AI 建议建立在真实运行证据之上。
2. 用统一数据模型连接需求、测试、证据、Issue、提案和回归。
3. 保留用户最终审批权，避免无审查的自主修改。
4. 使用离线 Mock Provider 保证课程演示可重复。
5. 保留跨语言、跨应用形态的扩展架构。

## 8. 局限性

- 浏览器执行器目前只完成动作计划和基础 HTTP 探测。
- 需求质量判断仍以规则为主，存在保守提示。
- 云端模型传输层尚未正式接入。
- 审批状态尚未持久化到 SQLite。
- 补丁只提供预览，不自动应用和执行回归。

## 9. 后续工作

1. 接入 Playwright，执行点击、输入、等待、断言和截图。
2. 捕获控制台错误和网络请求失败。
3. 使用模型对 Issue 进行源码定位并生成更准确的候选补丁。
4. 增加报告持久化、审批界面和 Git patch 应用。
5. 扩展 CLI、API 和桌面应用测试执行器。

## 10. 结论

SpecProbe 已经实现从需求理解到运行取证，再到问题报告和修复提案的基础闭环。实验表明，Rust 适合承担这类工具的 CLI、进程控制、结构化数据和安全边界管理。虽然当前浏览器执行深度和模型接入仍有限，但项目已经形成清晰、可测试、可扩展的工程基础，并能够完整展示 AI 代码审查助手的核心思路。
